# causalRisk → cleanTMLE integration report

Targeted additions for event-process classification, censoring and
competing-risk coherence, weight diagnostics, target-population
clarity, missing-data planning, cumulative-risk reporting, and
hazard-ratio de-emphasis. The cleanTMLE estimator API is unchanged.

## New functions added

| Function | Purpose |
|---|---|
| `clean_event_process_table()` | Structured classification of post-baseline events with role in estimand, ICH E9(R1) strategy, primary and sensitivity handling. Warns when censoring lacks a handling specification, when death is treated as censoring without justification, or when the same event is both competing event and censoring without an estimand justification. |
| `clean_check_event_processes()` | Coherence check: at each time point, sum of event-of-interest and competing-event cumulative incidences should equal the composite risk within tolerance; flags `OK`, `INCOHERENT`, `COMPONENT_EXCEEDS_COMPOSITE`, or missing components. |
| `clean_weight_diagnostics()` | Standard diagnostic summary for treatment, censoring, or missingness weights: percentiles overall and by treatment, ESS overall and by group, count and proportion of extreme weights, and (when covariates are supplied) standardised mean differences before and after weighting. Returns `flags` list for low ESS, extreme weights, and high max weight. |
| `clean_target_population()` | Forces explicit declaration of target population (full eligible / treated / untreated / overlap / trial-eligible / positivity-supported restricted / user-defined), reference group, restriction rule, rationale, and implications for interpretation. Warns on positivity-supported population without a rule and on treated/untreated population without an explicit ATT/ATU label. |
| `clean_missing_data_plan()` | Records baseline / treatment / outcome missingness, censoring / loss to follow-up, structurally undefined outcomes, and linkage failure as a single table with timing, presumed mechanism, primary handling, sensitivity handling, nuisance model used, identification assumption, and notes. |
| `clean_risk_report_table()` | Reporting table with one row per (time point × treatment strategy) cell, including event counts, censoring and competing-event counts, person time, ESS, risk and CI, risk difference, risk ratio, estimator, and nuisance specification. |

All six helpers carry a class (`cleantmle_event_process`,
`cleantmle_risk_check`, `cleantmle_weight_diag`,
`cleantmle_target_population`, `cleantmle_missing_plan`,
`cleantmle_risk_report`) with a print method registered in NAMESPACE.

## Tests added

`tests/testthat/test-event-process-helpers.R` (12 test_that blocks):

- `clean_event_process_table` returns required columns and `cleantmle_event_process` class;
- death-as-censoring without justification triggers a warning;
- censoring without handling triggers a warning;
- competing-event marked as censoring without justification triggers a warning;
- coherent component risks pass `clean_check_event_processes()`;
- incoherent component risks (component sum > composite) trigger `COMPONENT_EXCEEDS_COMPOSITE`;
- `clean_weight_diagnostics()` ESS matches `sum(w)^2 / sum(w^2)`;
- SMDs (unweighted and weighted) are computed and finite;
- low ESS triggers the `low_ess` flag;
- positivity-supported population without restriction rule warns;
- treated population without reference group warns;
- `clean_missing_data_plan()` distinguishes baseline / outcome / censoring;
- `clean_risk_report_table()` has required columns and tolerates missing optional fields;
- error on missing required field.

All twelve pass on a clean install. Full package suite: `..S......W.............W..........S........S..........` (no failures; 3 unrelated skips for optional packages; 2 expected warnings from warning-trigger tests).

## RescueCo sections added

After "Implications for Interpretation":

1. **Event-process classification and estimand implications.** Calls
   `clean_event_process_table()` to classify the GOSE outcome, death,
   loss to follow-up, inter-facility transfer (intercurrent event;
   excluded from primary cohort), administrative end of follow-up, and
   treatment discontinuation (not applicable for a one-time prehospital
   exposure). Includes a paragraph explaining which events are part of
   the estimand, which are censoring or missing-outcome processes, and
   which require sensitivity analyses.
2. **Target population and reference group.** Declares the full
   eligible direct-from-scene ambulance population with non-Rescue.Co
   transport as the reference, and notes that the inter-facility-
   transfer-inclusion sensitivity exhibit corresponds to a different
   target population.
3. **Weight, overlap, and positivity diagnostics.** Calls
   `clean_weight_diagnostics()` on the IPTW weights with a 10-max
   threshold and 0.4 N ESS floor, conditional on `ps_diag` being in
   scope.
4. **Cumulative-risk reporting.** Builds a `clean_risk_report_table()`
   from the existing `comparison` data frame.
5. **Sensitivity analyses mapped to assumptions.** Compact table mapping
   exchangeability, positivity, consistency, dependent censoring,
   missing outcome data, and competing/truncating events to the
   prespecified diagnostic or sensitivity analysis.

## Limitations and items requiring manual review

- The rescueCo case-study chunks for weight diagnostics and the
  cumulative-risk table use `eval = exists(...)` guards. They render
  whenever the upstream rescueCo pipeline has produced the relevant
  RDS objects in the same session (`ps_diag`, `comparison`); on a
  clean clone without those artefacts, the chunks are silently
  skipped and the rendered case-study HTML will show the helper
  prose without the printed tables. Re-running stages 1–4 of the
  rescueCo pipeline on cleanTMLE 0.1.1 before rendering the case
  study restores the full tables.
- `clean_check_event_processes()` performs an arithmetic coherence
  check; it does not estimate competing-risk cumulative incidences.
  Sites that need event-specific risk estimates should compose this
  helper with a survival-TMLE or Aalen–Johansen estimator (`survtmle`
  or `lmtp`).
- The supplementary table `tbl-event-diag` is intentionally short
  (six rows). Sites that want a longer mapping should expand it
  rather than fold it into the main regulatory-alignment table.
- We did not run `R CMD check`. The full test suite passes locally on
  R 4.4.2 with the existing dependencies.
- The vignette `library(cleanTMLE)` call is inside an
  `include = FALSE` setup chunk so it does not appear in the rendered
  output. Sites that build the vignette without the package
  installed will see a clear `library(cleanTMLE)` failure rather
  than a downstream "function not found" error.

  `run_tmle_targeting_step`, `extract_tmle_estimate`, `run_iptw_workflow`,
  `run_match_workflow`, `run_matched_tmle`, or `run_ipcw_tmle`.
- Numerical results in the rescueCo case study or the simulation
  study.
- The figures in the manuscript or vignette.
- The package version. The release-notes line for these helpers will
  be added to `NEWS.md` at the next version bump.
